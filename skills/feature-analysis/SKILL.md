# Feature Analysis — 需求分析到 Spec

**Announce at start:** "I'm using the feature-analysis skill 来完成需求分析，产出 spec.md、plan.md 和 tasks.md。"

**目标：** 本会话结束时，三个文件落盘到 `docs/specs/NNN-feature-name/`。实现工作在新会话里用 `feature-implement` 完成。

**三个文件的分工：**
- `spec.md` — 产品视角：功能描述、用户故事、业务规则、edge case
- `plan.md` — 工程视角：技术架构、数据结构、接口清单、核心逻辑
- `tasks.md` — 执行视角：每个任务只改一个文件，按依赖顺序执行

**发现业务特例时随时写入 MEMORY.md，不要等到最后。**

---

## Step 0: 加载项目上下文（必做，不可跳过）

先读，再想，不要凭空发散。

```
1. 读 CLAUDE.md（技术栈、禁止项、代码规范、模块边界）
2. 读 MEMORY.md（环境信息、业务特例、历史踩坑）
3. 如果需求涉及特定模块，读该模块的 CLAUDE.md / README.md
```

读完后确认：
- 这个需求涉及哪些模块？
- 有没有已知的相关禁止项或业务特例？
- 有没有相关的历史 bug 记录？

**上下文越精准，后续分析越不需要猜。**

---

## Step 1: 规模判断

| 规模 | 判断标准 | 路径 |
|------|----------|------|
| 小 | 单文件修改、无跨模块、无 DB 变更、无新 API | 跳过 Step 2-4，直接 Step 5 |
| 中 | 1-2 个模块、有 API 变更或 DB 变更 | 走完整流程 |
| 大 | 3+ 模块、新功能、状态机变更、多角色权限 | 走完整流程，plan.md 必做 |

---

## Step 2: 需求澄清（小改动跳过）

提出 **5 个关键澄清问题**，聚焦在：

1. 功能边界（做什么 / 明确不做什么）
2. 权限角色（哪些角色可以操作，行为是否有差异）
3. 状态机（涉及哪些状态，状态转换规则）
4. 并发与幂等（重复操作的预期行为）
5. 异常处理（失败时的回滚/补偿策略）

**等用户回答后，立即将澄清结果写入 spec.md 的"背景"章节（文件不存在则先创建），再进入 Step 3。**

```markdown
## 背景（澄清结果）
- 功能边界：[用户回答]
- 权限角色：[用户回答]
- 状态机：[用户回答]
- 并发与幂等：[用户回答]
- 异常处理：[用户回答]
```

**澄清结果必须写入 spec.md，不能只留在对话里——会话关闭后对话内容丢失。**

---

## Step 3: 发散理解（小改动跳过）

用 `superpowers:brainstorming` skill，基于已读的项目上下文和用户回答：

- 这个需求的真实意图是什么？
- 有哪些实现方向？哪个最符合现有架构？
- 哪些假设需要验证？
- 有没有和现有功能冲突的地方？

---

## Step 4: Edge Case 审计（小改动跳过）

**不要凭空列 edge case，先搜代码。禁止推断——搜索结果为空时必须标注"未找到代码证据，需人工确认"。**

搜索规则（防止上下文爆炸）：
- 只搜 tasks.md 里当前功能涉及的模块目录
- 文件类型限制：`*.go` / `*.ts` / `*.tsx`
- 每个关键词最多取前 20 条结果，超出截断
- 搜索关键词：状态字段名、枚举值、权限相关函数名

搜完后，列出矩阵（**超过 15 条说明功能过大，建议拆分**）：

```
边界情况矩阵（最多 15 条）：
- 状态码和状态转换路径（含特殊状态，标注代码位置）
- 权限角色的行为差异（来自搜索结果，未找到标注"需人工确认"）
- 并发场景下的幂等性要求
- 异常状态下的处理逻辑
```

**澄清结果和矩阵必须写入 spec.md 的"背景"章节，不能只留在对话里。**

**给出完整矩阵，用户确认后再进入 Step 5。**

---

## Step 5: 创建目录 + Agent 分工生成文件

查看 `docs/specs/` 下已有编号，取下一个，创建目录 `docs/specs/NNN-feature-name/`。

**三个文件用 Agent 分工生成，顺序不能颠倒（后者依赖前者）：**

```
Step 4 完成（edge case 矩阵已确认）
        ↓
  Agent A → spec.md（产品视角，不读代码）
        ↓ spec.md 落盘后
  Agent B → plan.md（工程视角，读代码 + 参考 spec.md）
        ↓ plan.md 落盘后
  主会话 → tasks.md（基于 plan.md 的文件路径）
```

---

### Agent A — 生成 spec.md

**角色：** Product Manager
**输入：** 需求描述 + 澄清结果 + edge case 矩阵
**不需要读代码**

prompt 模板：
```
你是 Product Manager，生成 spec.md 保存到 docs/specs/NNN-feature-name/spec.md。

需求描述：[用户原始需求]
澄清结果：[Step 2 的用户回答]
Edge Case 矩阵：[Step 4 的完整矩阵]

输出结构（严格按此顺序）：
# NNN - 功能名称 Spec

## 功能概述
一句话说清楚做什么。

## 背景（澄清结果）
- 功能边界：[来自 Step 2]
- 权限角色：[来自 Step 2]
- 状态机：[来自 Step 2]
- 并发与幂等：[来自 Step 2]
- 异常处理：[来自 Step 2]

## 用户故事
### US-1：场景名
**作为** 角色，**我希望** 操作，**以便** 目的。

## 影响范围
| 模块 | 变更类型 | 说明 |
（变更类型：DB/API/逻辑/UI）
> 技术架构、API 设计、DB 变更详见 plan.md

## 业务规则
权限矩阵、状态机、边界条件。

## Edge Cases
来自矩阵的完整内容，不要删减。

## 不在范围内
**此章节由用户填写或确认，不由 AI 自行决定。**
Agent A 生成草稿后，主会话必须展示给用户确认：
"以下内容不在本次范围内，请确认或补充：[列表]"

只写产品视角，不写任何技术实现细节。

完成后输出摘要：
- 用户故事数量
- 业务规则关键点（状态机、权限矩阵）
- Edge Case 数量
- 不在范围内的关键排除项
```

**Agent A 完成后，主会话做两步验证：**

**第一步：目标确认（防止 spec 偏离原始需求）**

向用户展示以下内容，等待确认：
```
## 请确认需求目标

功能概述：[spec.md 的功能概述]

用户故事：
- US-1：[一句话摘要]
- US-2：[一句话摘要]

不在范围内：
- [列表]

以上是否符合你的预期？如有偏差请指出。
```

**用户确认后才能继续。** 发现偏差时，在主会话里直接修正 spec.md，不重跑 Agent A。

**第二步：格式验证**
- 用户故事是否覆盖了所有澄清结果里的场景
- Edge Case 矩阵是否完整（对照 Step 4 的矩阵逐条检查）
- 有没有把技术细节写进 spec（如有，要求 Agent A 删除）

---

### 对抗验证 — Challenger Agent（spec 确认后必做）

spec 用户确认通过后，用一个独立 Agent 专门找问题：

```
你是资深需求分析师，你的任务是挑战这份 spec，找出所有问题。

读取：docs/specs/NNN-feature-name/spec.md

从以下四个角度逐条审查，每个问题必须有具体依据：

## 1. 业务规则自洽性
- 有没有相互矛盾的业务规则？
- 状态机是否完整（所有状态转换都有定义）？
- 权限矩阵是否有遗漏的角色或操作？

## 2. Edge Case 完整性
- 有没有明显遗漏的边界情况？
- 并发场景是否考虑到？
- 异常状态（网络超时、数据不存在、权限不足）是否都有处理？

## 3. 范围蔓延风险
- "不在范围内"章节是否足够明确？
- 有没有模糊的描述可能导致实现时范围扩大？

## 4. 与现有系统的冲突
- 新功能是否可能破坏现有接口契约？
- 是否依赖了尚未实现的功能？

输出格式：
## Challenger 报告
### 发现的问题
| 类型 | 问题描述 | 位置（哪个章节）| 严重程度（高/中/低）|

### 建议修改
[每个高/中严重度问题的具体修改建议]

### 结论
通过 / 需要修改（列出必须修改的问题）
```

**Challenger Agent 完成后：**
- 高严重度问题：必须修改 spec.md 后才能继续
- 中严重度问题：和用户确认是否需要处理
- 低严重度问题：记录到 spec.md 的"已知限制"章节，不阻塞流程

---

### Agent B — 生成 plan.md

**角色：** Software Architect
**输入：** spec.md + CLAUDE.md + 相关模块代码
**必须先读 spec.md 和 CLAUDE.md**

prompt 模板：
```
你是 Software Architect，生成 plan.md 保存到 docs/specs/NNN-feature-name/plan.md。

先读：
- docs/specs/NNN-feature-name/spec.md
- CLAUDE.md（技术规范和禁止项）
- 相关模块代码（[涉及的模块路径]）

**前端基座强制读取（不可跳过）：**
- 读 src/glite-ui/src/routes/routes.ts，提取：路由注册方式、文件引用格式（.tsx 还是 .vue）
- 读至少一个现有同类页面目录（如 src/glite-ui/src/pages/delivery-manage/），提取：文件命名规范、目录结构约定
- tasks.md 的前端文件路径必须与提取结果完全一致，禁止套用其他框架模板

输出结构（每个章节必须写到可直接指导实现的粒度）：

## 技术架构
精确到目录路径，例：
- Handler：模块/api/internal/handler/领域/
- Logic：模块/api/internal/logic/领域/
- 路由前缀：从现有路由文件确认（不要猜，grep 一条已有路由）
- 前端页面：src/pages/功能名/（从 routes.ts 和现有页面目录提取的命名规范）
- 前端 API：src/service/功能Api.ts（从现有 service 文件确认命名规范）

## 数据结构
直接写 struct/interface 代码（含枚举值说明），不写"类型定义"这种空话。
数据库表用字段/类型/说明三列表格。

## 接口清单
方法/路径/鉴权/说明四列表格，鉴权列居中。

## 核心逻辑设计（有状态机/算法/并发控制时必写）
状态机流转、算法伪代码、并发控制策略。

## 架构决策
写结论，不写检查清单：
- 接口契约：是否破坏现有调用方，如何兼容
- 权限隔离：本功能的租户隔离方式（特例查 MEMORY.md）
- DB 兼容性：迁移 SQL 的版本约束
- 跨模块边界：涉及哪些子模块，各自职责

遵守 CLAUDE.md 的代码规范和禁止项。

完成后输出摘要：
- 涉及的模块和文件路径列表
- 接口数量和路由前缀
- DB 表变更列表
- 与 spec.md 业务规则的对应关系（每条业务规则对应哪个技术实现）
```

**Agent B 完成后，主会话做 spec-plan 对齐验证：**
- spec 的每条业务规则在 plan 里都有对应实现（对照 Agent B 的摘要）
- spec 的状态机和 plan 的数据结构里的状态字段一致
- spec 的权限矩阵和 plan 的架构决策里的权限隔离策略一致
- plan 的数据结构字段名和类型没有和 spec 的业务规则冲突

**发现不一致时：** 在主会话里直接修正 plan.md，不要重新跑 Agent B。

---

### 主会话 — 生成 tasks.md

Agent B 完成且 spec-plan 对齐验证通过后，主会话读 plan.md，生成 tasks.md：

```markdown
# NNN - 功能名称 Tasks

> 每个任务只改一个文件。同层并行，跨层串行，每层完成后做跨层对齐验证再 commit。

## 完整度检查点

> feature-implement 每层完成后对照此表验证，所有层完成后做最终验收。

### spec 覆盖
- [ ] US-1 [场景名] → T?, T?
- [ ] US-2 [场景名] → T?
- [ ] Edge Case: [描述] → T?
- [ ] Edge Case: [描述] → T?

### plan 覆盖
- [ ] 接口 `METHOD /api/path` → T?
- [ ] 接口 `METHOD /api/path` → T?
- [ ] DB 表 [表名] → T?

---

## DB 迁移

- [ ] **T1** `模块/migrations/xxx.sql`
  - 变更描述

## 后端

- [ ] **T2** `模块/路径/文件名.go`
  - 功能描述
  - 路由：`METHOD /api/path`

- [ ] **T3** `模块/路径/文件名.go`
  - 功能描述
  - 路由：`METHOD /api/path`

## 前端

- [ ] **T4** `模块/路径/文件名.ts`
  - 功能描述
```

**生成 tasks.md 后验证：**
- 完整度检查点的 spec 覆盖：每个用户故事和 edge case 都有对应 task 编号
- 完整度检查点的 plan 覆盖：每个接口和 DB 表都有对应 task 编号
- tasks 里的路由和 plan.md 接口清单完全一致
- 已存在的文件路径用 Glob 确认真实存在（新建文件除外）
- 层级顺序正确（DB 迁移 → 后端 → 中间服务 → 前端）

**任务粒度：每个 task 只改一个文件。**
同一层级多个 task 互不依赖，feature-implement 会并行处理。

---

## Step 6: 一致性检查

三个文件都落盘后，检查：

- spec 的所有用户故事在 tasks 里都有对应 task
- spec 的 edge case 在 plan 的核心逻辑里有处理
- plan 的接口清单和 tasks 的路由一致
- tasks 顺序符合依赖关系（DB 迁移在后端之前）
- 有没有遗漏的模块

**实现方案一致性检查（必做，防止 spec/tasks 双路线）：**
- spec 的"影响范围"或"架构说明"里定义的实现方案（iframe-only / API 自研 / 混合）是否与 tasks 的具体文件和内容一致；若 spec 写 iframe，tasks 里不能出现 API 封装 task
- spec 里的主键口径（如按 `code` 查询而非 `id`）是否在 tasks 的接口描述里被明确体现
- spec 里的数据源口径（如查预聚合月表而非明细表）是否在 tasks 的 logic 描述里被明确体现
- 发现任何冲突，必须在本步骤修正 tasks.md，不能留到实现阶段

---

## 完成标准

- [ ] `docs/specs/NNN-feature-name/` 目录已创建
- [ ] spec.md：有用户故事、业务规则、完整 edge case 矩阵
- [ ] **用户已确认目标（功能概述 + 用户故事摘要）**
- [ ] **用户已确认"不在范围内"章节**
- [ ] plan.md：技术架构到目录级、数据结构有代码、接口有四列表格
- [ ] tasks.md：完整度检查点已填写（每个 US/edge case/接口都有对应 task 编号）
- [ ] tasks.md：顶部加需求冻结标记（见下）
- [ ] 业务特例已写入 MEMORY.md
- [ ] 告知用户：用 `feature-implement` skill 开始实现

**tasks.md 顶部需求冻结标记：**
```markdown
> ⚠️ 需求已冻结（[日期]）。实现阶段的需求变更必须走 change-request skill，不能直接改代码。

## 当前进度
- 分析阶段：✓ 完成
- DB 层：⬜ 未开始
- 后端层：⬜ 未开始
- 前端层：⬜ 未开始
```

**分析完成后关闭本会话，实现在新会话里进行。**

# Feature Refactor — 旧功能改造

**Announce at start:** "I'm using the feature-refactor skill 来执行功能改造。"

**核心原则：改造前必须知道影响范围，不能盲目修改。**

**发现业务特例时立即写入 MEMORY.md。**

---

## Step 0: 加载上下文

```
1. 读 CLAUDE.md（禁止项、代码规范）
2. 读 MEMORY.md（业务特例、历史踩坑）
3. 确认改造的功能名称和文件路径
```

---

## Step 1: 影响分析（Impact Analyzer）

**角色：资深架构师**
**目标：在改动任何代码之前，识别完整的影响范围。**

```
你是资深架构师，分析以下改造的影响范围：

改造目标：[功能名称 / 文件路径]
改造内容：[要改什么]

必须先读：
- CLAUDE.md（了解模块边界和禁止项）
- 目标文件及其直接依赖

用 Grep/Glob 分析（每个关键词最多 20 条结果）：
1. 调用链：谁调用了这个功能/接口？
   grep -r "函数名/接口路径" src/ --include="*.go" --include="*.ts" -l
2. 数据依赖：哪些地方读写了相关字段/表？
   grep -r "字段名/表名" src/ --include="*.go" -l
3. 接口契约：这个接口的 Request/Response 结构被哪些地方引用？

输出影响分析报告（保存到 docs/specs/NNN-refactor-name/impact-analysis.md）：

## 改造范围
[要改的文件和内容]

## 调用方列表
| 调用方文件 | 调用方式 | 影响等级（高/中/低）|

## 数据依赖
| 字段/表 | 读写位置 | 影响等级 |

## 接口契约变更
| 变更类型 | 字段 | 是否破坏性变更 |
（破坏性变更：删除字段、修改类型、修改枚举值）

## 风险评估
- 破坏性变更：[有/无，如有列出]
- 需要同步修改的文件：[列表]
- 建议的兼容策略：[如需要]
```

**Agent 完成后，主会话确认：**
- 有无破坏性变更（如有，必须制定兼容策略）
- 需要同步修改的文件是否都在 tasks.md 里
- 影响等级"高"的调用方是否有回归测试计划

**分析报告质量不足时的处理：**

| 问题 | 处理方式 |
|------|---------|
| 调用方列表不完整（Grep 结果有遗漏） | 主会话补充 Grep：`grep -r "函数名\|接口路径" src/ --include="*.go" --include="*.ts" -l`，手动补充到报告 |
| 破坏性变更未识别（Agent 漏判） | 主会话对照接口定义逐字段检查，直接修改 impact-analysis.md |
| 风险评估为空或过于乐观 | 标注"需人工确认"，在 spec.md 的"已知限制"章节记录，不阻塞流程 |
| 报告格式错误（缺少某个章节） | 主会话补充缺失章节，不重跑 Agent |

**禁止因报告不完整而重跑 Agent**——直接修正报告更高效。

## Step 2: 制定改造 Spec

基于 impact-analysis.md，用 `feature-analysis` skill 生成 spec.md + plan.md + tasks.md。

**传递给 feature-analysis 的额外输入：**
- 将 impact-analysis.md 的路径告知 Agent B（Software Architect），让其读取调用链和影响范围

**Agent B prompt 补充（在 feature-analysis 的 Agent B prompt 基础上追加）：**
```
额外输入：
- 影响分析报告：docs/specs/NNN-refactor-name/impact-analysis.md
  （读取调用方列表、接口契约变更、风险评估章节）

注意：
- 技术架构必须包含所有需要同步修改的调用方文件
- "架构决策"章节必须说明破坏性变更的兼容策略
- 接口清单必须包含所有受影响的接口（不只是改造目标）
```

**改造场景的特殊要求：**
- spec.md 的"不在范围内"章节必须明确：哪些调用方的行为不会改变
- tasks.md 必须包含所有需要同步修改的调用方文件（来自 impact-analysis.md）
- tasks.md 的完整度检查点必须覆盖所有调用方的回归验证

**Edge Case 审计时额外检查：**
- 改造后的接口是否向后兼容
- 现有调用方是否需要同步更新
- 数据库字段变更是否影响存量数据

---

## Step 3: 实现

用 `feature-implement` skill 按层级实现。

**改造场景的额外验证（每层完成后）：**
- 对照 impact-analysis.md 的调用方列表，确认所有调用方都已同步更新
- 对照接口契约变更，确认破坏性变更有兼容处理
- 用 Grep 确认没有遗漏的调用方

---

## Step 4: 回归验证

用 `qa-e2e` skill 验证，额外增加：

```
改造专项回归验证：
- [ ] impact-analysis.md 的所有调用方功能正常
- [ ] 破坏性变更有兼容处理，现有调用方不受影响
- [ ] 改造前后的核心业务流程行为一致
- [ ] 数据库存量数据不受影响
```

---

## 常见踩坑

| 问题 | 预防措施 |
|------|----------|
| 改了接口但漏掉某个调用方 | Step 1 影响分析必须用 Grep 找所有调用方 |
| 破坏性变更没有兼容处理 | Step 1 明确识别破坏性变更，Step 2 制定兼容策略 |
| 改了字段名但存量数据没处理 | Step 2 tasks.md 加数据迁移 task |
| 回归测试只测改动部分 | Step 4 必须测所有调用方 |
